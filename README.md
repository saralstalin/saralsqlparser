# SaralSQL Parser

SaralSQL is a high-fidelity, recursive-descent T-SQL parser built in TypeScript. Unlike standard SQL parsers designed for simple query execution, SaralSQL is architected specifically to power **Language Server Protocol (LSP)** features such as IntelliSense, precise error recovery, symbol renaming, and code navigation.

> [!IMPORTANT]
> **Status:** This package is currently in active development and powers the SaralSQL VS Code extension. It is not yet published as a standalone npm package.

---

## 🎯 The Goal

The primary objective of SaralSQL is to provide the structural and physical context required by modern IDEs (like VS Code and Azure Data Studio).

While many parsers discard "trivial" data like whitespace or exact offsets, SaralSQL preserves the relationship between the code and the document. This ensures that every AST node corresponds perfectly to the user's cursor position, enabling reliable editor tooling.

## ✨ Features

### Precision Tracking & Fault Tolerance

- **Byte-Perfect NodeLocation:** Every node — statements, expressions, join clauses, columns, scope boundaries — carries exact `start` and `end` byte offsets aligned to the source document.
- **Fault-Tolerant Parsing:** A `resync()` mechanism recovers from syntax errors and continues parsing the rest of the script. Errors are surfaced as `ErrorNode` entries in the AST rather than thrown exceptions.
- **Windows-Aware Lexing:** Dedicated `\r\n` handling prevents position drift in scripts generated by SSMS or exported from Windows environments.

### Core T-SQL Support

- **DML:** Full support for `SELECT` (including `TOP`, `DISTINCT`, subqueries, CTEs), multi-row `INSERT`, `UPDATE` with joins, and `DELETE` with alias-based join targets.
- **DDL:** `CREATE TABLE`, `VIEW`, `PROCEDURE`, `FUNCTION`, and `TYPE` including table-valued types and `IDENTITY` columns.
- **Joins:** All join types — `INNER`, `LEFT/RIGHT/FULL OUTER`, `CROSS JOIN`, `CROSS APPLY`, `OUTER APPLY` — with canonical type normalization and `rawType` preservation for round-trip fidelity.
- **CTEs:** Full `WITH` clause support including chained CTEs and `UNION`/`EXCEPT`/`INTERSECT` inside CTE bodies.
- **Control Flow:** `IF...ELSE` and `BEGIN...END` blocks parsed with correct T-SQL batch-scoping semantics.
- **Batch Support:** `GO` separators recognized and handled at both parse and scope level.

### Advanced Expression Engine

- **Pratt Parser:** Handles operator precedence correctly across arithmetic, bitwise, boolean, and T-SQL-specific operators (`BETWEEN`, `IN`, `LIKE`, `IS NULL`, `EXISTS`, `COLLATE`).
- **Window Functions:** `OVER (PARTITION BY ... ORDER BY ...)` with full span tracking.
- **Multipart Identifiers:** `[schema].[table]`, `#TempTable`, `@Variable`, and `dbo.Schema.Table` resolved into structured `MemberExpression` nodes.
- **Type-Safe Join Nodes:** `JoinType` is a TypeScript string literal union — exhaustive checking, no stringly-typed join logic.

### Scope & Symbol Layer

- **Hierarchical Scope Tree:** A `Scope` tree is built as a post-parse pass by `ScopeBuilder`, completely decoupled from parsing for performance.
- **Symbol Registration:** Variables (`DECLARE`), table variables, parameters, CTEs, tables, views, procedures, functions, and table/join aliases are all registered with kind, data type, and location.
- **Cursor Resolution:** `scope.findInnermost(offset)` resolves the innermost scope at any cursor position. `scope.getVisibleSymbols()` returns all symbols visible at that point, walking the parent chain.
- **Query-Local Aliases:** Table and join aliases are scoped per `SELECT` statement and do not leak into the outer scope.
- **Procedure Isolation:** Each `CREATE PROCEDURE`/`FUNCTION` body gets its own child scope. Parameters are registered locally and invisible to the batch.
- **Correct T-SQL Semantics:** Variables declared inside `IF` branches are batch-scoped (not block-scoped), matching actual T-SQL runtime behaviour.

## 🏗 Architecture

```
src/
├── lexer.ts          # Tokenizer — typed tokens with byte offsets
├── parser.ts         # Recursive-descent parser — produces typed AST
├── scope.ts          # Scope and Symbol model
└── scopeBuilder.ts   # Post-parse AST walk — builds Scope tree
```

**Lexer** tokenizes raw T-SQL into a stream of typed tokens. Keywords are normalized to uppercase at lex time for zero-cost comparison downstream. Module-level `Set` constants eliminate per-token allocations in the hot path.

**Parser** is a pure AST producer. It holds no scope state and performs no symbol resolution — its only job is transforming a token stream into a strongly-typed AST with embedded `NodeLocation` on every node.

**ScopeBuilder** walks the finished AST in a separate pass and builds the `Scope` tree. Decoupling this from parsing means the parser remains fast and the scope pass can be skipped, cached, or run incrementally by the caller.

**Scope** is fully encapsulated — internal `symbols` and `children` are private. The public API (`define`, `resolve`, `resolveLocal`, `findInnermost`, `getVisibleSymbols`, `getChildren`) is sufficient for all LSP consumers without exposing mutable internals.

## Why SaralSQL

There is no existing Node.js/TypeScript T-SQL parser with byte-accurate spans on every AST node and a scope model designed for LSP use. `node-sql-parser` treats T-SQL as one of fifteen dialects. The ANTLR4 T-SQL grammar produces a concrete syntax tree requiring a full transformation layer before it is useful for tooling. SaralSQL is built from the ground up for the one use case neither covers: reliable, cursor-aware T-SQL intelligence inside a code editor.
