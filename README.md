# SaralSQL Parser

SaralSQL is a high-fidelity, recursive-descent T-SQL parser built in TypeScript. Unlike standard SQL parsers designed primarily for query execution or AST transformation, SaralSQL is architected specifically for **Language Server Protocol (LSP)** workloads — IntelliSense, diagnostics, symbol resolution, rename, go-to-definition, code navigation, and editor-grade fault tolerance.

> **Status:** Production-grade parser core, actively developed and powering the SaralSQL VS Code extension.

## 🎯 The Goal

The primary objective of SaralSQL is to provide the structural, semantic, and physical context required by modern editors like Visual Studio Code and Azure Data Studio.

Most SQL parsers optimize for correctness on valid SQL. Editors need something different:

* SQL is often incomplete
* syntax is frequently broken while typing
* offsets must remain exact
* tooling must continue working even when parsing fails

SaralSQL is built around that reality.

Every AST node preserves exact source span information, and malformed SQL produces **recoverable partial AST nodes** rather than parser crashes.

## ✨ Features

## Precision Tracking & Fault Tolerance

* **Byte-Perfect NodeLocation**
  Every node — statements, expressions, joins, columns, aliases, scopes, and declarations — carries exact `start` and `end` byte offsets aligned to the source document.

* **Fault-Tolerant Parsing**
  A structured `resync()` mechanism recovers from syntax errors and continues parsing subsequent statements.

* **Recoverable AST Nodes**
  Broken statements become partial typed nodes (`incomplete: true`) rather than hard failures.

* **ErrorNode fallback**
  When recovery is impossible, parsing degrades into `ErrorNode` rather than throwing exceptions.

* **Windows-Aware Lexing**
  Dedicated `\r\n` handling prevents position drift in scripts generated by SSMS or exported on Windows.

### Continuation / Resync Safety

Recovery is **statement boundary safe**.

Broken SQL never swallows subsequent valid statements:

```sql
SELECT Name +
SELECT 1;
```

The second statement still parses normally.

Semicolon-safe recovery prevents runaway token consumption.
